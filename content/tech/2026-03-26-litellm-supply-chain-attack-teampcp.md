# 【資安警報】LiteLLM 供應鏈被駭！300萬日下載量陷危機｜開發者必須立刻行動

2026年3月，開源 AI 開發社群遭遇了近年來最嚴重的供應鏈攻擊事件。LiteLLM——這個每日下載量超過 300 萬次的熱門 Python 庫——在版本 1.82.7 和 1.82.8 中被植入惡意後門程式碼。攻擊者透過滲透 CI/CD 流程，成功取得開發者的敏感憑證。這起事件不僅震撼了 AI 開發圈，也再次提醒我們開源軟體供應鏈的脆弱性。

## 事件完整脈絡：TeamPCP 是如何發動攻擊的

這起攻擊的幕後主使者是名為 **TeamPCP** 的駭客組織。他們採用了極為精密的供應鏈滲透策略，目標並非直接攻擊終端用戶，而是透過污染開源軟體的開發流程，在軟體中植入可遠端控制的後門。

攻擊的關鍵切入點是 **Trivy**——一款廣泛使用的開源安全掃描工具。TeamPCP 首先滲透了 Trivy 的 CI/CD 流程，取得了其 PyPI 发布凭證。接著，他們利用這個信任基礎，向 PyPI 發布了被污染的 LiteLLM 版本 1.82.7 和 1.82.8。

由於 LiteLLM 在 AI 開發社群中的高人氣，加上許多開發者對 Trivy 這類安全工具的信任，被植入後門的版本迅速擴散開來。據估計，受影響的下載次數以百萬計，時間窗口長達數天。

## 受污染版本的分析：後門程式的惡意行為

安全研究人員很快對受影響版本進行了逆向分析。LiteLLM 1.82.7 和 1.82.8 中被植入的後門程式碼具有以下特徵：

**隱蔽性極高**：惡意程式碼被巧妙地嵌入在正常的函式庫初始化流程中，一般的程式碼審查難以發現。它使用了各種混淆技術來躲避靜態分析。

**遠端指令執行**：後門允許攻擊者在受感染的系統上執行任意 Python 程式碼。這意味著攻擊者可以竊取資料、安裝其他惡意工具，或是被利用作為進一步攻擊的跳板。

**憑證竊取**：最令人擔憂的是，該後門专门针对環境變數中的 API 金鑰和敏感憑證。許多開發者習慣將 OpenAI API 金鑰、AWS 憑證等存入環境變數，這些資料很可能已經被外洩。

## 攻擊技術細節：CI/CD 流程的薄弱環節

這起攻擊暴露了現代軟體開發流程中的嚴重安全漏洞。傳統上，我們認為只要從官方套件庫（如 PyPI）下載軟體就是安全的。但 TeamPCP 的攻擊證明了這個假設的錯誤。

攻擊者選擇攻擊 Trivy 的 CI/CD 流程，是因為：

1. **信任鏈的濫用**：開發者信任 Trivy 這個安全工具，攻擊者利用了這種信任。
2. **自動化流程的漏洞**：CI/CD 系統通常拥有较高的系统权限，一旦被攻破，影響範圍極大。
3. **發布流程的間接性**：從 Trivy 到 LiteLLM 的供應鏈關係較為間接，增加了攻擊的隱蔽性。

這也是典型的 **軟體供應鏈攻擊**（Supply Chain Attack）案例。攻擊者不需要直接攻擊目標，而是透過滲透上游供應商來達成目的。

## 開發者止血步驟：立即行動保護你的專案

如果你在近期使用過 LiteLLM，請立即執行以下步驟：

### 第一步：檢查版本

```bash
pip show litellm
```

如果版本顯示為 1.82.7 或 1.82.8，你的環境已經被感染。

### 第二步：升級到安全版本

```bash
pip install --upgrade litellm
```

請確保升級到 1.82.9 或更新的版本。官方已經緊急發布了安全補丁。

### 第三步：輪換所有憑證

這是最關鍵的一步。你必須假設所有存放在環境變數中的敏感資料都已經外洩。請立即：

- 重新產生所有 API 金鑰（OpenAI、Anthropic、AWS 等）
- 更新資料庫連線密碼
- 輪換 SSH 金鑰和部署 token
- 檢查雲端帳單和存取日誌，確認是否有異常活動

### 第四步：審計依賴項

使用以下工具掃描你的專案依賴項：

```bash
pip install pip-audit
pip-audit
```

此外，也建議檢查 Trivy 和其他可能受影響的工具是否為最新版本。

## 對 AI 開源生態系的長期警示

這起事件對整個 AI 開發社群都有深遠的影響：

**依賴項管理的警覺**：我們必須重新審視對開源庫的信任。每個依賴項都是一個潛在的攻擊面。在評估引入新庫時，除了功能需求，也要考慮其安全記錄和維護狀況。

**零信任安全的必要性**：即使是从官方來源下載的軟體，也不應該完全信任。隔離的開發環境、憑證的定期輪換、網路分段等安全實踐應該成為常態。

**CI/CD 安全的重要性**：開發流程本身需要被保護。實施 CI/CD 流程的安全審計、限制發布權限、使用雙因素認證等都是必要的措施。

**自動化安全工具的局限性**：這次攻擊諷刺地利用了安全工具 Trivy 的信任，說明依賴单一工具並不足夠。安全的縱深防禦策略是必要的。

## 常見問題

### LiteLLM 是什麼？

LiteLLM 是一個統一的 LLM 介面庫，讓應用程式可以輕鬆切換不同的大型語言模型供應商，是 AI 開發基礎設施的重要元件。它支援 OpenAI、Anthropic、Azure、Google 等多種 LLM API，提供標準化的介面，大幅簡化了多模型應用的開發工作。

### 我該如何檢查是否受影响？

立即執行 `pip show litellm` 檢查版本是否為 1.82.7 或 1.82.8。如果是，升級到最新安全版本並輪換所有環境變數中的 API 金鑰和憑證。同時也建議檢查其他可能受影響的依賴項，如 Trivy 等工具。

### 攻擊者取得了哪些資料？

根據研究，後門程式會嘗試竊取環境變數中的敏感資料，包括 API 金鑰、資料庫密碼、雲端服務憑證等。強烈建議假設所有在受感染期間使用的憑證都已經外洩，並立即進行輪換。

### 未來如何防止類似攻擊？

建議採用以下安全實踐：1) 使用pip-compile 或 Poetry 锁定依赖版本；2) 實施依賴項的安全掃描；3) 隔離開發和生產環境；4) 定期輪換敏感憑證；5) 對 CI/CD 流程實施安全審計和監控。
